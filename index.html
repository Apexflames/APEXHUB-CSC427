Introduction to Compilers
Compiler · Assembler · Interpreter · Differences
Topic 1
1.1 What is a Compiler?

Textbook Definition (Louden): A compiler is a program that reads a program written in one language — the source language — and translates it into an equivalent program in another language — the target language.

Lecturer's Definition: The goal of a compiler is to translate a source program in some high level language into some other form called target program. The source program is a high-level programming language (a problem-oriented language), and the target program is a mixture of machine level instructions, information about hardware registers, system resources, and memory addresses. An important part of any compiler is the detection and reporting of errors.

🗣️ Plain English
A compiler is like a human translator who reads a French book (your C++ code) and produces a complete English version (machine code). The translation is done all at once before anyone reads it. Once translated, the English version (executable) runs directly without needing the translator present.
1.2 Three Types of Translators
TranslatorDefinitionInput → OutputReal-Life Analogy
CompilerTranslates high-level source program into target (usually machine/assembly) code all at once before executionC/Java code → Machine codeTranslating an entire book into another language before publishing it
AssemblerTranslates assembly language programs into machine code (binary instructions for the CPU)Assembly → Machine codeConverting shorthand notes into full sentences the CPU understands directly
InterpreterTakes both source program AND input data, translating and executing each statement one at a time. Translation and execution phases are one and the same.Python/JS code + input → Output directlyA live interpreter at a conference translating speech as it happens, sentence by sentence
💡 Compiler vs Interpreter Key Difference
A compiler translates the entire program first, then you run the output separately. An interpreter translates and executes line by line simultaneously — it needs to be present every time the program runs. That's why compiled programs (C, C++) run faster but take time to compile; interpreted programs (Python, JavaScript) start instantly but run slower.
1.3 Types of Compilers
TypeDescription
Single-Pass CompilerMakes one pass through source code. Faster but limited optimization.
Multi-Pass CompilerMakes multiple passes. Better optimization.
Cross CompilerRuns on one platform but produces code for a different platform (e.g., compiling for ARM processor on an x86 machine)
Just-In-Time (JIT) CompilerCompiles code at runtime (e.g., Java JVM). Hybrid between compiler and interpreter.
Bootstrapping CompilerA compiler written in the language it compiles (e.g., C compiler written in C)

2.2 The Preprocessor

Definition: The preprocessor is a compile-time software tool which is called before the compiler itself is being executed. It is text-based — it has no knowledge of the syntax and semantics of the language but instead treats the source code as pure text.

Preprocessing Operations:

(a) Inclusion of Header Files▼

In C++, #include <iostream.h> is replaced by the full contents of the iostream.h header file before compilation. This gives the compiler access to declarations it needs.

🗣️ Plain English
Like "copy-pasting" a standard document into your own document before printing. The preprocessor literally pastes the header file contents into your code.
(b) Macro Expansion — #define▼

A macro is a fragment of code given a name. Whenever the name is used, it is replaced by the code. Example:

C macro example
#define MAX_SIZE 10
int main() {
    int size = 0;
    size = size + MAX_SIZE;   // preprocessor replaces MAX_SIZE with 10
    printf("\n value = %d\n", size);
    return 0;
}
🗣️ Plain English
Like using "Ctrl+F and Replace All" — every occurrence of MAX_SIZE gets replaced with 10 before the compiler even sees the code.
(c) Conditional Compilation — #ifdef / #endif▼

Selecting lines of source code to be compiled or ignored. Begins with #ifdef and ends with #endif.

💡 Real-Life Example
Like having a document with sections marked "Print only if audience is engineers." The preprocessor only includes the relevant sections based on a flag set at the top.
(d) Inline Functions▼

Grouping statements together with inline keyword. When an inline function is called, the compiler replaces the function call with the actual body of the function — saving the overhead of a function call.

✅ Benefit
Saves the time taken when calling a function and prevents writing the same code multiple times.
2.3 The Linker

Definition: A linker combines object code (machine code that has not yet been linked) produced from compiling and assembling many source programs, as well as standard library functions and resources supplied by the operating system.

It resolves references in each object file to external variables and procedures declared in other files, and resolves symbolic references to produce either an executable or dynamic library.

🗣️ Plain English
Imagine building a house where different teams built the kitchen, living room, and bathroom separately. The linker is the construction team that connects all these separately built rooms together into one complete house (executable program).
2.4 The Loader

Definition: Compilers, assemblers and linkers usually produce code whose memory references are made relative to an undetermined starting location — this is called relocatable machine code. A loader calculates appropriate absolute addresses for these memory locations and amends the code to use these addresses so the program can actually execute in memory.

💡 Key Terms
Relocatable Code — Machine code that uses relative addresses (can be loaded anywhere in memory).
Absolute Machine Code — Code with fixed, actual memory addresses ready to execute.
Literal — A constant value that appears directly in code (e.g., the number 42, the string "hello").
Delimiter — A character that separates or terminates tokens (e.g., semicolons, commas, braces in C).

3.1 Overview — Analysis/Synthesis Model

The process of compilation is split into six phases, each of which interacts with a symbol table manager and an error handler. This is called the analysis/synthesis model of compilation:

  • Analysis phases are machine-independent — they understand the source language
  • Synthesis phases are machine-dependent — they produce the target code

3.3 Symbol Table Management

Definition: A symbol table is a data structure containing all the identifiers (names of variables, procedures, etc.) of a source program together with all the attributes of each identifier.

For variables: type, memory size, scope. For procedures/functions: number and type of arguments, method of passing each argument, return type.

💡 Purpose
The symbol table provides quick and uniform access to identifier attributes throughout the compilation process. Information is usually put in during lexical and/or syntax analysis phases.
3.4 Error Handling

Each of the six phases can encounter errors. On detecting an error the compiler must: report the error helpfully, correct the error if possible, and continue processing to find further errors.

Lexical Analysis
Tokens · Patterns · Lexemes · Regular Expressions · Finite Automata
Topic 4
4.1 What is Lexical Analysis?

Textbook (Louden): Lexical analysis is the process of recognizing tokens from the source file. A lexical analyzer (also called a scanner or lexer) operates by reading characters from the source file and recognizing strings of characters that belong together as individual units of meaning — called tokens.

Lecturer's Definition: Lexical Analysis is the process of breaking the input string into tokens. The part of a compiler that reads the input of characters and converts it into a stream of tokens is called the Lexical Analyser or Lexer or scanner or tokenizer.

🗣️ Plain English
Think of reading an English sentence. Before understanding it, you first recognize the individual words (tokens). Lexical analysis does the same — it reads raw characters and groups them into meaningful "words" for the compiler. It doesn't care about grammar yet — just identifies what each piece is.
4.2 Token, Pattern, Lexeme — The Key Three
TermDefinitionExample
TokenAn object that has a type and a value. A category of lexeme. E.g. keyword, identifier, number.int(2), arithoperator(+)
PatternA rule that specifies when a sequence of characters from the input constitutes a token. Described using Regular Expressions.A letter followed by letters/digits = identifier pattern
LexemeAn instance of a token — the actual character string from the source code that matches the pattern. It is the actual token value.The string if is a lexeme matching the keyword token if

4.4 Responsibilities of a Lexical Analyser (5 Key)
  1. Tokenization — Reading characters and grouping them into meaningful tokens (keywords, identifiers, literals, operators, punctuation)
  2. Whitespace & Comment Removal — Stripping out spaces, newlines, tabs and comments which are not passed to the syntax analyser
  3. Symbol Table Interaction — When an identifier is found, the lexer enters it into the symbol table with its attributes
  4. Error Detection — Detecting lexical errors such as invalid characters or malformed lexemes and reporting them
  5. Line Counting — Keeping track of line numbers to help produce meaningful error messages referencing the correct location in source code
4.5 How Identifiers Are Handled

When the lexical analyser encounters an identifier (a variable name, function name, etc.):

  1. It reads the character sequence that matches the identifier pattern (letter followed by letters/digits)
  2. It checks the symbol table to see if this identifier already exists
  3. If it does NOT exist: a new entry is created in the symbol table with the identifier's name and attributes (type, scope, etc.)
  4. If it DOES exist: the existing entry is retrieved
  5. The lexer returns a token of type id (identifier) with a pointer to the symbol table entry as its value
4.6 Regular Expressions (RE)

Definition (Louden): Regular expressions are a formal way of specifying patterns for strings. They are the standard notation for describing the structure of tokens and are used to define the rules of the lexer.

Formal RE Definition (Lecturer): Let I be a class of regular languages over some alphabet Σ. Regular expressions are defined as follows:

RULE 1 — Empty Set
∅ (Phi)
∅ is an element of I, denoting the empty set. The corresponding RE is ∅.
RULE 2 — Empty String
λ (lambda)
{λ} is an element of I with regular expression λ. Represents the empty string.
RULE 3 — Single Character
∀ a ∈ Σ, {a}
For every a in alphabet Σ, {a} is an element of I with regular expression a.
RULE 4a — Concatenation
(r₁r₂)
L₁L₂ is an element of I with regular expression (r₁r₂), representing the concatenation of both languages. Example: ab matches the string "ab".
RULE 4b — Union / Alternation
(r₁ + r₂) or (r₁ | r₂)
L₁|L₂ is an element of I with regular expression (r₁ + r₂), representing the union. Example: a|b matches "a" OR "b".
RULE 4c — Kleene Star (zero or more)
(r₁)*
L₁* is an element of I with regular expression (r₁)*. Represents zero or more repetitions. Example: a* matches "", "a", "aa", "aaa", ...
4.7 RE to Finite Automaton — NFA & DFA

Specifications are written using Regular Expressions. These are then converted to Finite Automata (FA) to actually perform the recognition. There are two types:

TypeFull NameKey Property
NFANon-deterministic Finite AutomatonCan have multiple transitions for the same input; has ε-transitions (transitions on empty string). Easier to construct from RE.
DFADeterministic Finite AutomatonExactly one transition per input symbol per state. No ε-transitions. Used in actual implementation.

Conversion Process: RE → NFA (Thompson's Construction) → DFA (Subset Construction) → Minimized DFA → Lexer code

q₀ = start (no input), q₁ = accepting state (valid identifier found), q₂ = error state

Syntax Analysis (Parsing)
CFG · BNF · EBNF · Parse Trees · Parsers
Topic 5
5.1 What is Syntax Analysis?

Textbook (Louden): Syntax analysis, or parsing, is the process of determining the syntactic structure of a program — constructing a parse tree or syntax tree that represents the grammatical structure of the token sequence produced by the scanner.

Lecturer's Definition: Syntax or Parser is the structure of a language. It is the second step — a compiler needs to find the structure in the stream of tokens it gets from the lexer using grammar rules. The formal method for describing syntax was developed by Noam Chomsky in the 1950s using the idea of context-free grammar (CFG).

🗣️ Plain English
If lexical analysis reads individual words, syntax analysis checks that those words form grammatically correct sentences. For example, "i = 20" is a valid assignment statement, but "= i 20" is not — even though all three tokens are valid. The parser checks the structure.
5.2 Context-Free Grammar (CFG)

Definition: A context-free grammar (CFG) consists of:

  • Terminal symbols — actual tokens from the language (keywords, operators, identifiers)
  • Non-terminal symbols — names for grammar structures (expr, stmt, factor)
  • Productions — rules of the form: Nonterminal → sequence of terminals and nonterminals
  • Start symbol — the top-level nonterminal (e.g., S or program)

CFG permits a single nonterminal on the left-hand side of productions, followed by a right-hand side of symbols or other structure names.

5.3 BNF and EBNF

BNF (Backus-Naur Form) was developed by John Backus with contributions by Peter Naur, as a notational system for describing context-free grammars.

EBNF (Extended BNF) expresses more clearly the repetitive nature of BNFs.

BNF SymbolMeaningEBNF SymbolMeaning
::=Consists of / is the same as (definition)=Is defined to be
< >The name of the structure being described (nonterminal)[x] / ?Optional — 0 or 1 instances of x
|"Or" or Alternation{x} / x*0 or more instances of x
( )Group things togetherx{x} / x+1 or more instances of x
'x'Terminal symbol in single quotes
5.4 What is a Parser?

Definition: A parser is a component of a compiler that takes a sequence of tokens and builds a parse tree (or determines that the input does not conform to the grammar — a syntax error). The parse tree is a hierarchical representation of the grammatical structure of the input.

💡 Parser Strategies
Top-down parsing: Starts from the start symbol (root) and works DOWN towards the leaves (terminals) by expanding nonterminals. Constructs derivations from root to leaves.

Bottom-up parsing: Starts from the leaves (input tokens) and works UPWARD towards the root by reducing right-hand sides of rules to their left-hand nonterminals. Constructs derivations from leaves to root.

6.2 Top-Down Parsing

Definition: Top-down parsers expand non-terminals to match incoming tokens and directly construct a derivation (they construct derivations and parse trees from the root to the leaves). Start at the start symbol and apply productions to work towards the terminal symbols.

🗣️ Plain English
Like reading a recipe from the title down. You start with "Make Jollof Rice" (start symbol) and keep breaking it down: "Make Jollof Rice" → "Wash rice + Cook sauce + Combine" → "Get 2 cups rice + water + ..." until you reach the actual ingredients (terminals).
6.3 Bottom-Up Parsing

Definition: Bottom-up parsers attempt to match an input with the right-hand sides of grammar rules. When a match occurs, the right-hand side is replaced (reduced) to the nonterminal on the left. So they construct derivations and parse trees from the leaves to the root.

💡 Plain English
Like assembling a puzzle bottom-up — you start with individual pieces (tokens from the input) and figure out which group of pieces forms a smaller picture (production rule), then combine groups into bigger pictures until you have the full picture (start symbol).
6.4 Side-by-Side Derivation Example

Grammar: S → AB, A → aA | ε, B → b | bB — Derive aaab

6.5 Backtracking

Definition: Backtracking is an approach used in top-down parsing where if one production rule fails to produce the expected string, the parser undoes its choices (backtracks) and tries an alternative production.

How Backtracking Works (Example):

Grammar: S → aAb, A → cd | ce — parse string aceb

  1. Try S → aAb: match 'a', now need to expand A
  2. Try A → cd: match 'c', then try 'd' but input has 'e' — FAIL, backtrack
  3. Back to A, try A → ce: match 'c', match 'e' — SUCCESS
  4. Match 'b' — SUCCESS. String aceb is accepted.
⚠️ Limitation of Backtracking
If the given grammar has more than one alternative for a nonterminal, the cost of backtracking is high. For a grammar with many alternatives, the parser may need to try all of them before finding the correct one — making it exponential in the worst case. That's why predictive parsers (no backtracking) are preferred.
6.6 Issues in Parsing Context-Free Grammars
IssueDefinitionExampleSolution
Left RecursionA grammar is left-recursive if a nonterminal A can derive a string beginning with A: A → Aα. Top-down parsers go into infinite loops with left recursion.A → Aa | b (loops forever in top-down)Eliminate left recursion by rewriting: A → bA', A' → aA' | ε
Left FactoringWhen two or more productions for the same nonterminal begin with the same symbols, the parser cannot decide which to choose without looking ahead.A → αβ | αγ (both start with α)Factor out common prefix: A → αA', A' → β | γ
AmbiguityA grammar is ambiguous if a string can be derived by more than one parse tree. Ambiguous grammars cause parsing issues.E → E+E | E*E | id (2+3*4 has two parse trees)Rewrite grammar to enforce precedence and associativity
6.7 Substitution Example — Identifying and Eliminating Issues

Grammar: A → da | acB, B → abB | daA | Af

Step 1 — Identify issues:

  • B → Af — when A is substituted, A → da gives B → daf (OK), A → acB gives B → acBf — this introduces indirect left recursion through B → acBf

Step 2 — Substitute A into B:

  • Replace A in B → Af: using A → da: B → daf; using A → acB: B → acBf
  • Now B → abB | daA | daf | acBf

Step 3 — Identify and eliminate left recursion in B:

  • B → acBf introduces hidden recursion. Rewrite: group non-recursive terms as B → abB | daA | daf + B' where B' handles acBf
  • Result: B → (abB | daA | daf) B', B' → acBfB' | ε

Semantic Analysis
Type Checking · Symbol Table · Declarations · Types
Topic 7
7.1 What is Semantic Analysis?

Textbook (Louden): Semantic analysis is concerned with computing additional information about a program that is beyond what can be expressed in a context-free grammar. The primary purpose is to check for type consistency and determine the meaning of program constructs.

Lecturer's Definition: A semantic analyser takes its input from the syntax analysis phase in the form of a parse tree and a symbol table. Its purpose is to determine if the input has a well-defined meaning; in practice semantic analysers are mainly concerned with type checking and type coercion based on type rules.

🗣️ Plain English
Lexical analysis checks spelling of words. Syntax analysis checks grammar of sentences. Semantic analysis checks MEANING. "Colourless green ideas sleep furiously" is grammatically correct but semantically meaningless. Similarly, adding a string to a number is syntactically valid but semantically wrong — the semantic analyser catches this.

Parsing only verifies that the program consists of tokens in a syntactically valid combination. Semantic analysis identifies the operators and operands of expressions and statements from the hierarchical structure determined by the syntax analysis phase. It checks for:

  • Type errors
  • Compatibility of operators and operands
  • Undefined variables
  • Index out of bounds
  • Types of arguments to function calls
7.2 Types and Declarations

A type is a set of values and a set of operations operating on those values.

Type CategoryDescriptionExamples
Base TypesPrimitive types provided directly by the underlying hardwareint, float, double, char, bool
Compound TypesTypes constructed as aggregations of base types and simple compound typesarrays, pointers, records, structs, unions, classes
Complex TypesAbstract data types — may or may not have language supportlists, stacks, queues, trees, heaps, tables

A declaration is a statement in a program that communicates to the compiler the name and type of a data object. In many languages it may include modifiers that control visibility and lifetime.

7.3 Type Checking

Definition: Type checking is the process of verifying that each operation executed in a program respects the type system of the language — that all operands in an expression are of appropriate types and number.

TypeDefinitionExample
Static Type CheckingDone at compile-time. Defines restrictions on the structure of valid texts. Checks that identifiers are declared before use and used in appropriate context.C, Java — compiler rejects int x; x = "hello";
Dynamic Type CheckingDone at runtime (execution). Defines how constructs should produce program behaviour. A type error halts execution.Python — x = 5; x = "hello" is allowed, type is determined at runtime
7.4 Three Formal Methods of Semantic Analysis
  • Operational Semantics — Describes the meaning of a program by specifying how statements are executed on an abstract machine (step by step)
  • Denotational Semantics — Describes meaning by mapping programs to mathematical objects (functions). Meaning is denoted mathematically.
  • Axiomatic Semantics — Based on formal logic. Describes meaning of programs using logical assertions (preconditions and postconditions). Used in program verification.

Intermediate Code, Optimization & Code Generation
TAC · Optimization Types · Code Generation
Topic 8
8.1 Intermediate Code Generation

After the analysis phases are completed, the source program is decomposed into a symbol table and a parse tree. From this, the compiler generates intermediate code according to one of two approaches:

  1. Generate code for a specific machine, or
  2. Generate code for a 'general' or abstract machine, then use further translators to convert to specific machine code — this approach is more modular and efficient

The most widely used intermediate language is Three-Address Code (TAC) — a sequence of optionally labelled instructions where each instruction has at most three addresses (two operands and one result).

Three-Address Code Example for: k = i + j
t1 = i + j       ; compute i + j, store in temp t1
k  = t1          ; assign result to k
8.2 Code Optimization

Textbook (Louden): An optimizer attempts to improve the time and space requirements of a program. Optimization is a program transformation technique which tries to improve the code by making it consume less resources (CPU, memory) and deliver high speed.

Three Rules a Code Optimizer Must Follow:

  1. The output code must not, in any way, change the meaning of the program
  2. Optimization should increase the speed of the program and if possible, demand less resources
  3. Optimization should itself be fast and not delay the overall compiling process
Optimization TypeDescription
Peephole OptimizationOperates on target assembly — considers a few instructions at a time (through a "peephole") and makes simple machine-dependent improvements. E.g. eliminating multiplication by 1, removing redundant loads.
Local OptimizationPerformed by looking only at the statements in a basic block. Requires little analysis. Examples: constant folding, dead code elimination.
Global OptimizationApplies optimizations across basic blocks using data-flow analysis. More complex. Also called "intraprocedural methods".
Loop OptimizationActs on statements making up a loop (e.g. loop invariant code motion). Significant impact — many programs spend most time in loops.
Machine-IndependentCompiler transforms intermediate code without involving CPU registers or absolute memory locations
Machine-DependentDone after target code generated. Involves CPU registers. Takes maximum advantage of memory hierarchy.
🗣️ Plain English
Optimization is like editing a first draft of an essay. You find redundant sentences (remove them), combine short sentences (reduce code), and rearrange paragraphs for better flow — but the essay still says the SAME THING. The meaning never changes, just the efficiency.
8.3 Code Generation

The final phase of the compiler generates code for a specific machine. In this phase the compiler considers:

  • Memory management — allocating variables and data structures in memory
  • Register assignment — deciding which values to keep in CPU registers for speed
  • Machine-specific optimisation — exploiting special CPU instructions

The output from this phase is usually assembly language or relocatable machine code.

Q1(b). When it encounters identifiers, how does it handle them?▼

When the lexical analyser encounters an identifier (e.g., i, j, k, main, printf):

  1. The lexer reads the character sequence that matches the identifier pattern: a letter followed by zero or more letters or digits
  2. It checks the symbol table to see if this identifier was previously recorded
  3. If the identifier does NOT exist in the symbol table: a new entry is created with the identifier's name, leaving its attributes (type, scope, value) to be filled in later by syntax/semantic analysis
  4. If the identifier DOES exist: the existing symbol table entry is retrieved (no duplicate created)
  5. The lexer returns a token of type id along with a pointer/reference to the symbol table entry as the token's value
  6. This pointer is what allows subsequent compiler phases to access information about the identifier
💡 Why this matters
Identifiers cannot be turned into machine code directly — the compiler needs to know their type, memory location, and scope first. The symbol table is where all this information is collected across phases.

Q1(e). State five (5) responsibilities of a lexical analysis phase▼
  1. Tokenization — Reading the source character stream and grouping characters into meaningful tokens (keywords, identifiers, literals, operators, punctuation symbols)
  2. Whitespace and Comment Removal — Stripping out spaces, tabs, newlines and comments, which carry no semantic meaning and are not passed to the syntax analyser
  3. Symbol Table Population — When an identifier is encountered for the first time, the lexer enters it into the symbol table. This stores the identifier's name and provides a reference for future compiler phases
  4. Lexical Error Detection and Reporting — Detecting and reporting invalid character sequences (lexical errors), such as an invalid character (\$), an incomplete string literal, or a malformed number
  5. Line and Column Tracking — Keeping track of source file line numbers and column positions so that error messages produced by the compiler (in any phase) can accurately reference their location in the original source code

Finite Automaton for Identifier: letter(letter|digit)*

States: q₀ (start), q₁ (accepting), q₂ (error)

  • q₀ on letter → q₁ (first character is a letter — valid start)
  • q₁ on letter → q₁ (continue reading letters — stay in accepting state)
  • q₁ on digit → q₁ (continue reading digits after first letter — valid)
  • q₀ on digit → q₂ (identifier can't start with a digit — error)
  • q₁ on other → token complete, return id token

q₁ is a double-circled accepting state — whenever we are in q₁ and the next character is no longer letter/digit, we have successfully recognized an identifier.

Q2(b). Transform the RE into a Finite State Machine: (i) a*((bb)⁺ + bc*)   (ii) ((a*|b)cd)*▼

(i) a*((bb)⁺ + bc*)

Breakdown: a* = zero or more a's. (bb)⁺ = one or more pairs of bb. bc* = b followed by zero or more c's. The full RE matches either path after the a's.

FSM Construction:

States: q₀→q₁→q₂ (path for bb+) and q₀→q₃ (path for bc*)

  • q₀: start state, read a's (self-loop on a)
  • q₀ on b → q₁ (begin (bb)⁺ path) or q₃ (begin bc* path) — NFA has ε-choice here
  • q₁ on b → q₂ (completed first bb)
  • q₂ on b → q₁ (loop for additional bb pairs — this gives (bb)⁺)
  • q₂ = accepting state (one or more bb completed)
  • q₃: after first b, read c's (self-loop on c), q₃ is also accepting (bc* matched)

(ii) ((a*|b)cd)*

Breakdown: (a*|b) = either zero-or-more a's, OR a single b. Then cd = literal c then d. The whole thing * = zero or more times.

FSM Construction:

  • q₀: start/accepting (Kleene star allows zero repetitions)
  • q₀ on a → q₁ (reading a's) or q₀ on b → q₂
  • q₁ on a → q₁ (more a's), q₁ on c → q₃
  • q₂ on c → q₃
  • q₃ on d → q₀ (completed one (a*|b)cd, loop back)
  • q₀ is both start and accepting (handles zero repetitions of the group)

Q3(a). Define: delimiter, loader, linker, relocatable code, lexeme, literal▼
TermDefinition
DelimiterA character or sequence of characters that marks the boundary between tokens or program units. In C/C++, delimiters include semicolons (;), commas (,), braces ({, }), and parentheses. They separate or terminate syntactic constructs.
LoaderA system program that loads an executable file into memory for execution. It calculates appropriate absolute addresses for memory locations (which are stored as relative in relocatable code) and adjusts the code to use these actual addresses before execution begins.
LinkerA program that combines multiple object code files (produced by compiling/assembling separate source files) along with library routines into a single executable. It resolves symbolic references between object files — connecting a function call in one file to its definition in another.
Relocatable CodeMachine code in which memory references are expressed relative to an undetermined starting address rather than fixed absolute addresses. This allows the code to be loaded and executed at any location in memory. Produced by assemblers and compilers before linking and loading.
LexemeAn instance of a token — the actual sequence of characters from the source code that matches a token pattern. For example, the lexeme if matches the keyword token pattern if. It is the concrete character string, while the token is the abstract category.
LiteralA constant value that appears directly in source code — it has a fixed, directly-specified value. Examples: the integer literal 42, the string literal "hello", the floating point literal 3.14, the boolean literal true.
Q3(b). Explain the phases of a compiler design▼

The process of compilation is split into six phases, each interacting with a symbol table manager and an error handler:

  1. Lexical Analysis (Scanner): Reads the source program character by character and groups characters into tokens. Strips whitespace and comments. Enters identifiers into the symbol table. Output: a stream of tokens.
  2. Syntax Analysis (Parser): Takes the token stream and checks that tokens are arranged in a grammatically correct structure according to the language's context-free grammar. Builds a parse tree (syntax tree). Output: a parse tree.
  3. Semantic Analysis: Checks the parse tree for semantic correctness — primarily type checking. Verifies that operators are applied to compatible operands, variables are declared before use, and function arguments match parameter types. Output: annotated parse tree.
  4. Intermediate Code Generation: Translates the annotated parse tree into an intermediate representation (such as Three-Address Code). This form is machine-independent and easier to optimize. Output: intermediate code.
  5. Code Optimization: Improves the intermediate code to run faster and use less memory, while preserving the program's meaning. Common optimizations: removing dead code, loop optimization, constant folding. Output: optimized intermediate code.
  6. Code Generation: Translates the optimized intermediate code into target machine code (or assembly language). Handles register allocation, memory management, and machine-specific optimizations. Output: assembly/machine code.

Additionally: Symbol Table Management and Error Handling interact with all six phases throughout the process.

Q4(a). What is a parser? Generate parse tree for "aabcde" from S → aABe, A → Abc | b, B → d. Differentiate strategies.▼

What is a Parser?

A parser is a compiler component that takes a sequence of tokens (from the lexer) and determines its grammatical structure according to the language's context-free grammar. It builds a parse tree — a hierarchical tree representation of the derivation of the input string from the grammar's start symbol. If the input does not conform to the grammar, the parser reports a syntax error.

Generating Parse Tree for string "aabcde" from: S → aABe, A → Abc | b, B → d

Using top-down (leftmost) derivation:

  1. S → aABe (apply S → aABe)
  2. aABe → aAbcBe (apply A → Abc to leftmost A)
  3. aAbcBe → aabcBe (apply A → b to inner A)
  4. aabcBe → aabcde (apply B → d)
  5. Result: aabcde ✓

Q4(b). Backtracking approach to parse "aceb" from S → aAb, A → cd | ce▼

Backtracking is the top-down strategy where if a chosen production fails, the parser undoes and tries the next alternative.

Parsing "aceb" from S → aAb, A → cd | ce:

  1. Apply S → aAb: Match input[0]='a' with 'a' — SUCCESS. Now need to match A against remaining "ceb".
  2. Try A → cd: Match 'c' against input[1]='c' — SUCCESS. Match 'd' against input[2]='e' — FAIL ('d' ≠ 'e'). Backtrack to A.
  3. Try A → ce: Match 'c' against input[1]='c' — SUCCESS. Match 'e' against input[2]='e' — SUCCESS. A has been matched.
  4. Back in S → aAb: Match 'b' against input[3]='b' — SUCCESS.
  5. All input consumed. String "aceb" is ACCEPTED.
⚠️ Backtracking Cost
Backtracking requires re-scanning already-consumed input. For grammars with many alternatives, the cost grows exponentially. This is why LL(1) parsers (which look one token ahead to choose the correct production without backtracking) are preferred for practical use.
Question 5 — 15 Marks
Q5(a). Discuss the issues in parsing context free grammar▼

The main issues encountered when parsing context-free grammars are:

  1. Left Recursion: A grammar is left-recursive if a nonterminal A can derive a string starting with itself: A ⟹* Aα. Top-down parsers get into infinite loops with left-recursive grammars because they keep trying to expand A by applying A → Aα, producing another A to expand.
    Solution: Eliminate left recursion by rewriting: if A → Aα | β, rewrite as A → βA', A' → αA' | ε
  2. Left Factoring: Occurs when two or more productions for the same nonterminal share a common prefix. The parser cannot decide which production to use without knowing more of the input.
    Example: A → αβ₁ | αβ₂ — the common α must be "factored out."
    Solution: A → αA', A' → β₁ | β₂
  3. Ambiguity: A grammar is ambiguous if a single string can be derived in two or more different ways (producing different parse trees). Ambiguity causes issues because the compiler cannot determine the unique intended structure.
    Example: E → E+E | E*E | id produces two trees for "id+id*id" depending on operator precedence.
    Solution: Rewrite grammar with explicit precedence rules.
  4. Empty Productions (ε-productions): Productions of the form A → ε can complicate parsing algorithms, particularly in computing FIRST and FOLLOW sets for LL(1) parsers.
  5. Indirect Left Recursion: Occurs when A ⟹* Aα through a series of productions (not directly). Harder to detect and eliminate.
Q5(b). Show issues in A → da | acB, B → abB | daA | Af and eliminate them▼

Given grammar: A → da | acB    B → abB | daA | Af

Step 1 — Identify the issues:

Looking at B → Af: This production has A on the right-hand side. When A is substituted, A can derive acB (which starts with a), and acB then appears in B's production — this creates indirect left recursion because B can eventually derive strings that lead back to B.

  • B → Af → (da | acB)f → daf | acBf
  • B → acBf introduces: B derives a string containing B — indirect left recursion through substitution

Step 2 — Substitute A into B → Af:

Replace B → Af using A → da and A → acB:

  • From A → da: B → daf
  • From A → acB: B → acBf

Updated grammar for B: B → abB | daA | daf | acBf

Step 3 — Identify left recursion in updated B:

B → acBf — this is not direct left recursion (doesn't start with B), but B appears inside, making it right-recursive with a trailing f. The grammar now generates strings correctly.

Step 4 — Check for left-recursion in B:

None of the productions start with B, so there is no direct left recursion. The indirect recursion from B → Af → acBf has been eliminated by substitution.

Final Grammar:

  • A → da | acB
  • B → abB | daA | daf | acBf

The original issue (B → Af which caused indirect left recursion via A → acB → acBf) has been eliminated by substituting A into the production.
